### Required

**Verify cluster is not using a Kubernetes reconciler**

If your cluster has an existing version of {{prodname}} installed, verify that the cluster is not managed by any kind of Kubernetes reconciler. For example, if addon-manager exists, there will be an annotation called, `addonmanager.kubernetes.io/mode` on either of the following resources (if the resources exist):

- `tigera-operator` deployment in the `tigera-operator` namespace
- `calico-node` daemonset in the `kube-system` namespace

**Verify IAM permissions**

Verify your user account has IAM permissions to create Kubernetes ClusterRoles, ClusterRoleBindings, Deployments, Service Accounts, and Custom Resource Definitions. The easiest way to grant permissions is to assign the "Kubernetes Service Cluster Admin Role‚Äù to your user account. For help, see [AKS access control](https://docs.microsoft.com/en-us/azure/aks/control-kubeconfig-access)

**Review {{prodname}} requirements**

- [Network requirements](requirements.mdx#network-requirements) to ensure network access is properly configured for {{prodname}} components.

- If using a private registry, familiarize yourself with this guide on [using a private registry](private-registry/index.mdx)

- Ensure that you have the [credentials for the Tigera private registry and a license key](calico-enterprise.mdx)
